1. ABSTRACT (DRAFT)
====================================================================

Physical access-control systems remain a critical security layer in modern organizations, yet traditional rule-based approaches often fail to detect subtle insider-threat behaviors that imitate legitimate access patterns. This thesis presents RAPTORX, a full-stack anomaly-detection system that combines operational access-control workflows with machine-learning-based risk scoring. The system integrates a FastAPI backend, a Next.js operations dashboard, and an ensemble decision engine composed of Isolation Forest and Autoencoder models.

RAPTORX uses engineered behavioral features derived from badge-access events and applies a semi-supervised anomaly-detection strategy in which models are trained on normal behavior distributions. The final runtime signal is generated through weighted score fusion of Isolation Forest and Autoencoder outputs. A three-tier risk policy (Grant/Delay/Deny) converts model output into actionable operational decisions.

The implementation includes synthetic-data generation, leakage-aware preprocessing, threshold tuning, model versioning, structured audit logging, and thread-safe concurrent inference. Evaluation on synthetic hold-out data indicates strong detection capability; however, because training and evaluation are conducted on simulated access data, this thesis explicitly frames real-world pilot validation and threshold recalibration as necessary deployment steps.

Keywords: physical access control, anomaly detection, ensemble learning, isolation forest, autoencoder, synthetic data, security monitoring

2. METHODOLOGY
====================================================================

2.1 System Context
RAPTORX is designed for real-time physical access decision support. The system ingests access events, engineers risk-relevant features, evaluates anomaly likelihood using an ensemble model, and emits one of three outcomes: granted, delayed, or denied. The architecture combines rule constraints and machine-learning signals to support operationally interpretable outcomes [CITATION].

2.2 Dataset Generation
Due to the limited public availability of labeled physical badge-access datasets, this research employs synthetic data generation to construct controlled and reproducible experiments [CITATION]. RAPTORX uses scripts/generate_data_fixed.py as the primary data generator.

Default generation configuration in the current codebase:
- Total records: 500,000
- User profiles: 500
- Default anomaly ratio (development profile): 7%
- Production profile anomaly ratio: 1.5%
- Feature schema: 19 engineered features plus binary label

The generator includes seven anomaly categories and role-aware normal behavior simulation. This design introduces realistic class overlap and mitigates trivial class separation effects that can inflate reported performance in synthetic settings [CITATION].

2.3 Data Splitting and Leakage Control
Data is split using stratified sampling to preserve anomaly prevalence across subsets. The current default split is:
- Training set: 80%
- Test set: 20%

Preprocessing uses MinMaxScaler fitted on training data only, which prevents statistical leakage from evaluation subsets [CITATION]. RAPTORX maintains dual scaler artifacts:
- scaler_13.pkl for runtime model input schema (13 features)
- scaler_19.pkl for analytics workflows (19 features)
- scaler.pkl as legacy alias to scaler_13.pkl

For threshold tuning, RAPTORX uses data/processed/val_scaled.csv when available; otherwise, a validation subset is stratified from train_scaled.csv.

2.4 Model Design
RAPTORX applies a semi-supervised ensemble anomaly strategy:

1) Isolation Forest:
- Trained on normal-only records
- Detects structural outliers in feature space

2) Autoencoder:
- Trained on normal-only records
- Uses reconstruction error as anomaly signal

3) Ensemble fusion:
- Combined risk = 0.3 × IF score + 0.7 × AE score

This design balances complementary strengths of tree-based and reconstruction-based anomaly detection [CITATION].

2.5 Decision Policy
Runtime decision thresholds in the backend are environment-configurable with defaults:
- Grant threshold: 0.30
- Deny threshold: 0.70

Decision mapping:
- Risk < 0.30 -> GRANTED
- 0.30 <= Risk < 0.70 -> DELAYED
- Risk >= 0.70 -> DENIED

If models are unavailable, RAPTORX uses rule-based fallback logic to preserve service continuity.

2.6 Reliability and Concurrency Controls
RAPTORX includes explicit thread-safety mechanisms in the decision engine:
- Class-level initialization lock for one-time model loading
- Instance-level re-entrant lock for prediction critical sections
- Lock-minimized design for stateless operations

Repository validation artifacts report 220 concurrent decisions with zero race-condition errors and zero deadlocks.

5. LIMITATIONS, ETHICS, AND DEPLOYMENT CONSIDERATIONS
====================================================================

5.1 Synthetic Data Limitation
Although RAPTORX uses behaviorally rich simulation, synthetic distributions may not fully represent evolving real-world human behavior, operational exceptions, or adversarial adaptation [CITATION].

5.2 Transferability and Recalibration
Thresholds and score distributions learned on synthetic data should be recalibrated under real deployment conditions, ideally during a logging-only pilot phase before enforcement.

5.3 Governance and Human Oversight
Production use should include policy controls for:
- Auditability and access to decision records
- Data retention and privacy compliance
- Human review for high-impact denial outcomes
- Bias monitoring across worker groups/schedules
